4. Undo & Recover
CommandExplanation
git restore <file>Discard changes in working directory.
git restore --staged <file>Unstage a file.
git reset --soft <commit>Move HEAD (keep changes staged).
git reset --mixed <commit>Move HEAD (keep changes unstaged).
git reset --hard <commit>Move HEAD (discard all changes).
git reset --soft HEAD~1Undo last commit, keep changes.
git reset --mixed HEAD~1Undo last commit, unstage changes.
git reset --hard HEAD~1Undo last commit, discard changes.
git revert <commit>Create new commit to undo changes.
git reflogShow reference log.
Use Cases
Restore file changes:git restore <file>
Unstage file:git restore --staged <file>
Undo last commit (keep):git reset --soft HEAD~1
Undo last commit (discard):git reset --hard HEAD~1
See history of actions:git reflog

7. Stash (Save Work in Progress)
CommandExplanation
git stashStash (save) your changes.
git stash push -m "message"Stash with a message.
git stash listList all stashes.
git stash showShow changes in latest stash.
git stash show -pShow changes in latest stash (patch).
git stash applyApply latest stash (keep stash).
git stash popApply latest stash and remove it.
git stash dropDelete a stash.
git stash clearDelete all stashes.
When to Use Stash?
•Switching branches quickly.
•Pulling latest changes.
•Half-done work.
•Trying something new.
Example
git stash
git switch feature-branch
git pull
git stash pop

🍴 Fork + PR Workflow (No Write Access)
When to use: You found a repo (public or private with read-only access) and want to contribute
Step-by-Step Commands
Step 1Click Fork button on GitHub (top-right)
Step 2git clone https://github.com/YOU/REPO.git
Step 3git remote add upstream https://github.com/OWNER/REPO.git
Step 4git remote -v
Step 5git checkout -b my-feature
Step 6git add . && git commit -m "msg"
Step 7git push -u origin my-feature
Step 8Open Pull Request on GitHub UI
After git remote -v you should see:
origin https://github.com/YOU/REPO.git
upstream https://github.com/OWNER/REPO.git
🔀 Fork vs Collaborator — Which Workflow?
Situation Collaborator (Direct) Fork + PR
Do you have write access? ✅ Yes ❌ No (read-only / public)
Clone which repo? Original repo Your fork
Push where? git push origin branch git push origin branch
(to your fork)
Need PR? Optional (depends on team) ✅ Always required
Add upstream remote? ❌ Not needed ✅ Required
Who merges? You (if allowed) or team Original repo owner
Private repo? Owner invited you as collaborator Need at least read access to fork
Public repo? Rare (need invite) ✅ Standard open source way
The Golden Rule
✅ Direct Collaborator
git clone ORIGINAL
git checkout -b feature
git push origin feature
→ Open PR from branch in same repo
🍴 Fork Workflow
git clone YOUR_FORK
git remote add upstream ORIGINAL
git checkout -b feature
git push origin feature
→ PR from your fork to original
🔄 Keep Fork Up-to-Date + PR Tips
Syncing your fork with the original repo (important!)
CommandWhy
git fetch upstreamDownload latest changes from original.
git checkout mainSwitch to your local main branch.
git merge upstream/mainMerge original's main into yours.
git push origin mainUpdate your fork on GitHub too.
Or use rebase instead of merge:
git fetch upstreamDownload updates.
git rebase upstream/mainRebase your branch on top of original.
Good Pull Request Tips
✅One feature / one fix per PR — keep it small.
✅Write a clear title and description.
✅Sync with upstream before opening PR.
✅Respond to review comments quickly.
❌Don't push to main directly on your fork if PR is open.
❌Don't force push after PR is reviewed.
PR States on GitHub
🟡 Open Waiting for review / merge.
🟢 Merged Your changes are now in the original.
🔴 Closed Rejected or closed without merge.
🟣 Draft Work in progress, not ready yet.
